# Part 4 — Prompts are Production Code

> *If it isn't in Git with a version tag, we're just "voodoo programming".*

## What's here

| File | What it does |
|------|--------------|
| [`prompt_loader.py`](./prompt_loader.py) | Parse comment-style metadata (`# version: 3.7`) from prompt.txt files. Edit prompts in place — Git tracks the history. |
| [`eval_runner.py`](./eval_runner.py) | CLI: run an eval set against a prompt, write scores.json, exit non-zero on regression. |
| [`prompts/loan_rejection_explanation/`](./prompts/loan_rejection_explanation/) | Reference prompt artifact: `prompt.txt` (versioned), `eval.jsonl` (test cases), `scores.json` (last run). |
| [`ci/run_evals.sh`](./ci/run_evals.sh) | Bash script that runs every prompt's eval set. Called from GitHub Actions on every PR. |
| [`ci/compare_versions.py`](./ci/compare_versions.py) | Diff two prompts' scores.json (across Git refs) to show regressions and improvements. |

## Run

From the repo root:

```bash
python -m vol1_foundations.part04_prompts_as_code.eval_runner \
  vol1_foundations/part04_prompts_as_code/prompts/loan_rejection_explanation
```

Or the bash script:

```bash
./vol1_foundations/part04_prompts_as_code/ci/run_evals.sh \
  vol1_foundations/part04_prompts_as_code/prompts
```

## The shape of a prompt artifact

Every prompt is a directory with three files:

```
prompts/loan_rejection_explanation/
├── prompt.txt       # the prompt body, with metadata header
├── eval.jsonl       # test cases — one JSON object per line
└── scores.json      # auto-generated by eval_runner; gets committed
```

`prompt.txt` starts with comment-style metadata:

```
# version: 3.7
# updated: 2026-05-12
# author: priya.menon@company.com
# eval_pass_threshold: 0.85

You are a customer service writer for a digital lending platform...
```

Version lives **in the file**, not the filename. PRs show real word-
level diffs. CI re-runs evals on every change. Scores file gets
committed alongside the prompt change. That's the entire discipline.

## What this catches that "vibes-based" prompt engineering doesn't

- **Silent regressions.** You "improve" the prompt for one edge case
  and lobotomize 12 others. The eval set catches it before merge.
- **"Which version was running?"** Every API call logs the prompt
  version (see `common/llm_client.py`). At 2 a.m. when something
  breaks in production, this question is answerable in 30 seconds.
- **A/B testability.** Want to test v3.8 against v3.7? Branch the
  prompt file in a feature branch. CI runs evals against both. You see
  the diff in scores.json before deciding.

The eval runner uses an LLM judge under the hood — see
[`vol5_evals/part23_llm_judge/`](../../vol5_evals/part23_llm_judge/).
